Database Design Explanation — Order Management System
Overview

The database schema is designed for a basic Order Management System that handles customers, products, orders, and payments. The design follows normalization principles to reduce redundancy and maintain data integrity. Each table represents a specific entity, and relationships are defined using foreign keys.

Tables Explanation
1. Customers Table

Purpose:
Stores customer details required for placing orders.

Fields:

customer_id — Primary Key used to uniquely identify each customer.
name — Stores customer name.
email — Stores unique email address.
phone — Stores customer contact number.
created_at — Stores the date and time when the customer was added.

Reason for Design:
This table separates customer information from orders to maintain normalization and avoid duplicate customer data.

2. Products Table

Purpose:
Stores all product-related information.

Fields:

product_id — Primary Key.
product_name — Name of the product.
category — Product category.
price — Price of the product.

Reason for Design:
Keeping product details in a separate table allows multiple orders to reference the same product without duplication.

3. Orders Table (Core Table)

Purpose:
Stores order details placed by customers.

Fields:

order_id — Primary Key.
customer_id — Foreign Key referencing Customers.
order_date — Date when the order was placed.
status — Current order status.

Relationship:

One Customer → Many Orders

Reason for Design:
This structure allows tracking multiple orders placed by a single customer.

4. Order_Items Table (Bridge Table)

Purpose:
Stores product details associated with each order.

Fields:

order_item_id — Primary Key.
order_id — Foreign Key referencing Orders.
product_id — Foreign Key referencing Products.
quantity — Number of products ordered.

Relationship:

Orders ↔ Products (Many-to-Many)

Reason for Design:
An order can contain multiple products, and a product can be part of multiple orders. This table resolves the many-to-many relationship.

5. Payments Table

Purpose:
Stores payment information related to each order.

Fields:

payment_id — Primary Key.
order_id — Foreign Key referencing Orders.
amount — Payment amount.
payment_method — Payment type.
payment_date — Date of payment.

Relationship:

One Order → One Payment

Reason for Design:
Separating payment details improves financial tracking and system flexibility.

Relationships Summary

The database includes the following relationships:

Customers → Orders (One-to-Many)

Orders → Order_Items (One-to-Many)

Products → Order_Items (One-to-Many)

Orders → Payments (One-to-One)

These relationships ensure accurate tracking of orders and associated products and payments.

SQL Queries Explanation

Now we explain the 3 required queries — this is important for grading.

Query 1 — Top 3 Customers with Highest Number of Orders
SELECT 
    c.customer_id,
    c.name,
    COUNT(o.order_id) AS total_orders
FROM Customers c
JOIN Orders o 
    ON c.customer_id = o.customer_id
GROUP BY 
    c.customer_id, c.name
ORDER BY 
    total_orders DESC
LIMIT 3;

Purpose:
Identifies the top three customers who have placed the highest number of orders.

How It Works:

Joins Customers and Orders tables.
Counts orders per customer.
Sorts results in descending order.
Returns top 3 customers.

Business Use:
Helps identify loyal or high-value customers.

Query 2 — Retrieve Orders Placed in Last 30 Days
SELECT *
FROM Orders
WHERE order_date >= NOW() - INTERVAL 30 DAY;

Purpose:
Fetches recent orders placed within the last 30 days.

How It Works:

Filters orders using date comparison.
Returns only recent orders.

Business Use:
Useful for monitoring recent sales activity.

Query 3 — Calculate Total Revenue for Each Product
SELECT 
    p.product_name,
    SUM(oi.quantity * p.price) AS total_revenue
FROM Products p
JOIN Order_Items oi 
    ON p.product_id = oi.product_id
GROUP BY 
    p.product_name
ORDER BY 
    total_revenue DESC;

Purpose:
Calculates total revenue generated by each product.

How It Works:

Joins Products and Order_Items tables.
Multiplies quantity by price.
Sums total revenue per product.

Business Use:
Helps identify top-performing products.

Design Advantages

The database design provides:

Reduced redundancy
Improved data consistency
Support for multiple products per order
Accurate payment tracking
Scalable structure for future expansion
